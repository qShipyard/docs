This guide gets you from zero to a working narratorlog instance on your local machine. You will clone the repo, start the full stack with one Docker Compose command, complete the setup wizard, connect a real repository, and generate your first audience-specific changelog draft — all without writing a line of configuration.

<Note>
  The quickstart uses insecure placeholder secrets and plain HTTP. It is designed for local trials only. Do not expose this stack to the internet or use it for a real team deployment. See [Self-Hosting](/self-hosting) when you are ready to go to production.
</Note>

## Prerequisites

- Docker running on your machine (Docker Desktop or Docker Engine with Compose v2)

That's it. No Go, Node, Rust, or any other runtime required.

## Steps

<Steps>
  <Step title="Clone the repo and start the stack">
    Clone the narratorlog repository and run the quickstart Compose file from the repo root:

    ```shell
    git clone https://github.com/qShipyard/narratorlog.git
    cd narratorlog
    docker compose -f deploy/docker-compose.quickstart.yml up -d --build
    ```

    The first run downloads base images and compiles the API, worker, web app, and Rust reader — expect **several minutes** before the stack is ready. Subsequent starts are much faster.
  </Step>

  <Step title="Wait for the stack to become ready">
    Follow the logs to watch all services come up:

    ```shell
    docker compose -f deploy/docker-compose.quickstart.yml logs -f
    ```

    Once you see the web app and API reporting as ready, open [http://localhost:3000](http://localhost:3000) in your browser.
  </Step>

  <Step title="Complete the setup wizard at /setup">
    On a fresh database, narratorlog automatically redirects you to `/setup`. The wizard walks you through two things:

    1. **Create your workspace** — give your narratorlog instance a name
    2. **Create your admin account** — set your email and password

    This account becomes the owner of the workspace. You can invite additional team members later from the Team settings page.
  </Step>

  <Step title="Activate your instance at /activate">
    After setup, you land on `/activate` to connect the tools narratorlog needs to generate its first draft:

    1. **Connect your git provider** — paste a Personal Access Token for GitHub, GitLab, or Bitbucket. No OAuth app registration is required. The token needs read access to the repositories you want to scan.
    2. **Set up your AI provider** — enter an API key for Anthropic, OpenAI, or point narratorlog at a local Ollama instance.
    3. **Connect your first repository** — choose a repository from the list of repos your token can access.

    Once activation is complete, you land on the dashboard.
  </Step>

  <Step title="Trigger your first scan">
    From the dashboard, select your connected repository and click **Run scan** to trigger a manual scan. narratorlog will fetch recent commits, filter noise, group changes, and send them through the AI summarization pipeline.

    The scan progress appears in the dashboard. For a repository with recent commit activity, a standard scan typically completes in under two minutes.
  </Step>

  <Step title="Review the generated drafts">
    When the scan finishes, open the scan review page. You will see:

    - **Left panel:** the commit groups narratorlog identified — what actually shipped, grouped by pull request or inferred domain
    - **Right panel:** audience-specific drafts, tabbed by audience (developers, product, marketing, executives, and any others you configured)

    Read through each draft. You can edit any draft directly in the UI before approving it.
  </Step>

  <Step title="Approve and deliver">
    When you are satisfied with a draft, click **Approve**. Delivery is triggered automatically — the draft is sent to the output destinations you configured during activation (Slack channel, Notion page, email, and so on).

    Any draft you do not approve stays in a pending state until you return to it.
  </Step>
</Steps>

## Useful Docker commands

```shell
# Follow logs for all services
docker compose -f deploy/docker-compose.quickstart.yml logs -f

# Stop the stack (keeps your database intact)
docker compose -f deploy/docker-compose.quickstart.yml down

# Fresh start — wipes the database and runs first-run setup from scratch
# Use this if migrations fail or you want a clean slate
docker compose -f deploy/docker-compose.quickstart.yml down -v
docker compose -f deploy/docker-compose.quickstart.yml up -d --build
```
